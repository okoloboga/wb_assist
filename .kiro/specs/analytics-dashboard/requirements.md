# Requirements Document

## Introduction

Веб-дашборд для визуализации аналитики продаж и состояния складов Wildberries. Система предоставляет минималистичный интерфейс с инфографикой для отслеживания ключевых метрик бизнеса в реальном времени.

## Glossary

- **Dashboard**: Веб-интерфейс для отображения аналитических данных
- **Time Period**: Временной срез данных (день, неделя, месяц, год)
- **Order Metrics**: Метрики заказов (заказы, выкупы, отмены, возвраты)
- **Warehouse State**: Состояние складов (номенклатура, размеры, склады)
- **Chart Component**: Компонент визуализации данных (столбцовая диаграмма, круговая диаграмма)
- **Frontend Service**: Отдельный фронтенд-сервис на React/Vue/Next.js
- **Backend API**: Существующий API сервер для получения данных

## Requirements

### Requirement 1

**User Story:** Как пользователь, я хочу видеть метрики заказов в разных временных срезах, чтобы анализировать динамику продаж

#### Acceptance Criteria

1. WHEN пользователь открывает Dashboard, THE System SHALL отобразить четыре метрики заказов: заказы, выкупы, отмены, возвраты
2. WHEN пользователь выбирает временной период, THE System SHALL обновить данные метрик для выбранного периода
3. THE System SHALL поддерживать четыре временных среза: день, неделя, месяц, год
4. THE System SHALL отображать метрики в виде столбцовых или круговых диаграмм
5. WHEN данные загружаются, THE System SHALL показать индикатор загрузки

### Requirement 2

**User Story:** Как пользователь, я хочу видеть состояние складов, чтобы контролировать остатки товаров

#### Acceptance Criteria

1. THE System SHALL отображать информацию о складах под графиками продаж
2. THE System SHALL показывать номенклатуру товаров на складах
3. THE System SHALL отображать размеры товаров
4. THE System SHALL показывать названия складов
5. THE System SHALL визуализировать данные складов в виде таблицы или карточек

### Requirement 3

**User Story:** Как пользователь, я хочу использовать дашборд на мобильных устройствах, чтобы отслеживать метрики в любом месте

#### Acceptance Criteria

1. THE System SHALL адаптировать интерфейс для экранов шириной от 320px до 1920px
2. WHEN ширина экрана меньше 768px, THE System SHALL переключиться на мобильную компоновку
3. THE System SHALL сохранять читаемость графиков на мобильных устройствах
4. THE System SHALL обеспечивать удобную навигацию на сенсорных экранах
5. THE System SHALL оптимизировать размер шрифтов для мобильных устройств

### Requirement 4

**User Story:** Как пользователь, я хочу видеть актуальные данные, чтобы принимать обоснованные решения

#### Acceptance Criteria

1. WHEN Dashboard загружается, THE System SHALL запросить актуальные данные из Backend API
2. THE System SHALL обновлять данные при изменении временного периода
3. WHEN запрос к API завершается ошибкой, THE System SHALL отобразить сообщение об ошибке
4. THE System SHALL кэшировать данные для уменьшения количества запросов
5. THE System SHALL показывать время последнего обновления данных

### Requirement 5

**User Story:** Как разработчик, я хочу запускать фронтенд отдельно от бэкенда, чтобы упростить разработку

#### Acceptance Criteria

1. THE Frontend Service SHALL быть отдельным npm проектом
2. THE System SHALL запускаться командой npm run dev
3. THE System SHALL работать без Docker контейнеров в режиме разработки
4. THE System SHALL подключаться к Backend API через настраиваемый URL
5. THE System SHALL поддерживать hot reload при изменении кода

### Requirement 6

**User Story:** Как администратор, я хочу развернуть дашборд через nginx, чтобы обеспечить production deployment

#### Acceptance Criteria

1. THE System SHALL собираться в статические файлы командой npm run build
2. THE System SHALL быть готов к развертыванию через nginx
3. THE System SHALL поддерживать настройку API URL через переменные окружения
4. THE System SHALL включать конфигурационный файл nginx
5. THE System SHALL оптимизировать bundle size для production

### Requirement 7

**User Story:** Как пользователь, я хочу видеть минималистичный и понятный интерфейс, чтобы быстро находить нужную информацию

#### Acceptance Criteria

1. THE System SHALL использовать минималистичный дизайн без избыточных элементов
2. THE System SHALL применять единую цветовую схему для всех компонентов
3. THE System SHALL обеспечивать высокую контрастность для читаемости
4. THE System SHALL использовать понятные иконки и подписи
5. THE System SHALL группировать связанные элементы визуально
